## 1. 我对项目目标的理解

CyGES（Cycle Configuration Generation & Evolution System）的核心目标是：

1. 通过外部输入（热源、冷源、闭式循环边界、分位点等）构建系统对象；
2. 系统对象调用物性求解器生成拓扑与状态；
3. 在子循环层面计算做功与换热，并在系统层完成汇总；
4. 将子循环及外部流整理为换热匹配输入（最终用于 TQ 图与换热网络）；
5. 将部分变量（当前主要是子循环流量）暴露为优化变量，并在约束下求解。

一句话总结：  
**这是一个“拓扑生成 + 热力计算 + 换热匹配 + 优化接口”的闭式循环系统建模平台。**

## 2. 当前业务对象关系（我理解的主线）

### 2.1 系统层

- `SystemModel`
  - 包含多个 `ClosedCycle`
  - 包含多个外部流（热源/冷源）
  - 对外提供 `build()` 与 `solve()`

### 2.2 循环层

- `ClosedCycle`
  - 根据边界和分位点生成：
    - 节点（nodes）
    - 过程边（edges）
    - 子循环（subcycles）
  - 负责循环级统计（功率、换热）

### 2.3 子循环层

- `SubCycle`
  - 是后续优化最核心的单元
  - 包含：
    - 四角节点（raw / working）
    - 四条边（left/right/top/bottom）
    - `m_dot` 及上下界
  - 提供功热计算接口

### 2.4 求解层

- `CoolPropSolver -> CoolProp`
  - 提供 TP / PS 等状态求解
  - 在新框架内部统一工程单位输出

## 4. 需求拆解（面向实现）

基于你已给出的核心诉求，我将需求拆成以下可执行块：

### R1. 输入驱动建模

- 支持外部输入多组：
  - 热源过程
  - 冷源过程
  - 闭式循环边界
- 支持每个闭式循环独立分位点配置

### R2. 拓扑生成

- 从边界与分位点生成初始：
  - 节点
  - 边
  - 子循环
- 子循环流量提供默认值与可优化边界

### R3. 子循环热力计算

- 子循环根据左右边计算做功
- 根据上下边计算吸/放热
- 支持未来按边效率扰动节点状态（保留原始节点）

### R4. 系统级汇总

- 全部闭式循环聚合后，输出系统级：
  - 总功率
  - 总吸热
  - 总放热
  - 循环级明细

### R5. 换热匹配与 TQ

- 将子循环上下边 + 外部流统一整理为 TQ 段
- 为后续换热网络计算和优化准备接口

### R6. 优化接口

- 先暴露 `subcycle m_dot` 为主要优化变量
- 预留效率、边约束、温差约束等扩展口

## 5. 架构原则（用于后续约束代码）

1. **双轨数据原则**  
   `raw`（原始）不可覆盖，`working`（计算态）可重建。

2. **对象优先原则**  
   迁移阶段可以导出旧字典结构做兼容，但内部以对象关系为主。

3. **单位统一原则**  
   领域对象统一工程单位；求解器适配层内部处理 SI 转换。

4. **可优化友好原则**  
   每个优化变量都要有 `value/min/max/fixed` 语义位置。

5. **诊断可观测原则**  
   每步核心生成结果都有计数与异常统计，便于排错与对比。
